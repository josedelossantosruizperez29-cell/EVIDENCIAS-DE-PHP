El Patrón MVC

La arquitectura que separa datos, presentación y lógica para que tu código sea mantenible, escalable y profesional.

¿Qué es un patrón de diseño?

Un patrón de diseño es una solución probada y reutilizable a un problema recurrente en el desarrollo de software. No es código que copias: es una receta de estructura que guía cómo organizar tus clases, archivos y responsabilidades.

Reutilizable

El mismo patrón funciona en proyectos pequeños y grandes, en PHP, Java, Python…

Lenguaje común

Cuando dices "es MVC", cualquier desarrollador entiende la estructura sin ver el código.

Probado en batalla

No estás inventando algo nuevo — estás usando lo que miles de equipos ya validaron.

MVC (Model-View-Controller) es el patrón de diseño de arquitectura más utilizado en desarrollo web. Laravel, Symfony, Ruby on Rails, Django, ASP.NET… todos lo implementan.

Historia y origen

1979
Trygve Reenskaug (Xerox PARC) propone MVC para el lenguaje Smalltalk-80. El objetivo: separar la interfaz de usuario de la lógica de negocio.
2005+
Con el auge del desarrollo web (Ruby on Rails, Django, CodeIgniter), MVC se convierte en el estándar de facto para aplicaciones web. Laravel (PHP) lo populariza definitivamente en 2011.

Los 3 componentes en detalle

Model

El corazón lógico de la aplicación. Gestiona los datos y las reglas de negocio.

  • Consulta y escribe en la base de datos
  • Valida datos
  • Aplica reglas de negocio
  • No sabe nada de HTML ni de HTTP
Ejemplos:
Producto.php, Usuario.php, Pedido.php
View

La interfaz de usuario. Solo muestra datos, nunca los procesa.

  • Genera HTML para el navegador
  • Usa variables que le pasa el Controller
  • Puede tener lógica mínima (foreach, if)
  • Nunca consulta la BD directamente
Ejemplos:
lista.php, detalle.php, formulario.php
Controller

El director de orquesta. Recibe la petición y coordina M y V.

  • Recibe la petición HTTP ($_GET, $_POST)
  • Llama al Model para obtener/guardar datos
  • Pasa los datos a la View
  • No genera HTML ni consulta BD directamente
Ejemplos:
ProductoController.php, AuthController.php

Antes de MVC: el "código espagueti"

Sin MVC, un archivo PHP típico mezcla todo en el mismo lugar: consultas SQL, lógica de negocio y HTML. Esto se llama código espagueti porque, como los fideos, todo está enredado con todo.

Sin MVC — Todo mezclado
productos.php (antes)
<?php
// SQL directo aquí
$pdo = new PDO('mysql:host=localhost;dbname=tienda', 'root', '');
$stmt = $pdo->query('SELECT * FROM productos WHERE activo = 1');
$productos = $stmt->fetchAll();

// Lógica de negocio aquí
$descuento = 0.1;
foreach ($productos as &$p) {
    $p['precio_final'] = $p['precio'] * (1 - $descuento);
}
?>
<!DOCTYPE html>
<html>
<head><title>Productos</title></head>
<body>
<h1>Productos</h1>
<?php foreach ($productos as $p): ?>
    <div>
        <?php
        // Más lógica aquí dentro del HTML
        $clase = $p['stock'] > 0 ? 'disponible' : 'agotado';
        ?>
        <p class="<?= $clase ?>">
            <?= $p['nombre'] ?> - $<?= $p['precio_final'] ?>
        </p>
    </div>
<?php endforeach; ?>
</body>
</html>
  • ¿Quieres reutilizar la consulta? Copia y pega.
  • ¿Quieres cambiar el diseño? Buscas entre SQL y PHP.
  • ¿Quieres hacer pruebas? Imposible sin levantar una BD.
Con MVC — Cada cosa en su lugar
models/Producto.php
<?php
class Producto {
    // Solo datos y reglas de negocio
    public static function todos(): array {
        $pdo = Database::conexion();
        return $pdo->query('SELECT * FROM productos
                            WHERE activo = 1')
                   ->fetchAll();
    }
    public static function precioConDescuento(array $p): float {
        return $p['precio'] * 0.9;
    }
}
controllers/ProductoController.php
<?php
class ProductoController {
    // Solo coordina M y V
    public function index(): void {
        $productos = Producto::todos();
        View::render('productos/lista', compact('productos'));
    }
}
views/productos/lista.php
<!-- Solo HTML + variables -->
<h1>Productos</h1>
<?php foreach ($productos as $p): ?>
    <p><?= esc($p['nombre']) ?></p>
<?php endforeach; ?>

Estructura de directorios estándar

Así se organiza una aplicación PHP con MVC:

mi-app/ ├── index.php ← Front Controller │ ├── core/ │ ├── Router.php ← Enruta peticiones │ └── View.php ← Carga vistas │ ├── controllers/ ← C de MVC │ ├── HomeController.php │ └── ProductoController.php │ ├── models/ ← M de MVC │ └── Producto.php │ └── views/ ← V de MVC ├── layout/ │ ├── header.php │ └── footer.php └── productos/ ├── lista.php └── detalle.php
index.php — El Front Controller

Único punto de entrada. Todas las peticiones pasan por aquí. Su único trabajo es cargar las clases del core e iniciar el Router. Equivale a la puerta principal del restaurante.

core/Router.php

Lee la URL, decide qué Controller y qué método llamar. Ejemplo: ?c=producto&a=index → crea ProductoController y llama a index().

core/View.php

Método estático render($vista, $datos) que carga el header, la vista solicitada con las variables extraídas de $datos, y el footer.

Código de cada capa explicado

core/Router.php
core/Router.php
<?php
class Router
{
    public function despachar(): void
    {
        // Lee el controlador de la URL: ?c=producto  →  'producto'
        $nombreCtrl = ucfirst($_GET['c'] ?? 'home') . 'Controller';
        // Lee la acción: ?a=show  →  'show'
        $accion     = $_GET['a'] ?? 'index';

        $archivo = __DIR__ . '/../controllers/' . $nombreCtrl . '.php';

        // Si el controller no existe, redirige al 404
        if (!file_exists($archivo)) {
            $nombreCtrl = 'HomeController';
            $accion     = 'notFound';
            $archivo    = __DIR__ . '/../controllers/' . $nombreCtrl . '.php';
        }

        require_once $archivo;

        $controller = new $nombreCtrl();
        $controller->$accion();   // llama al método correcto
    }
}
core/View.php
core/View.php
<?php
class View
{
    /**
     * Carga el layout completo (header + vista + footer).
     *
     * @param string $vista  Ruta relativa en views/  ej: 'productos/lista'
     * @param array  $datos  Variables a exponer en la vista
     */
    public static function render(string $vista, array $datos = []): void
    {
        // extract() convierte ['key' => 'val'] en $key = 'val'
        // así la vista puede usar $key directamente
        extract($datos);

        require_once __DIR__ . '/../views/layout/header.php';
        require_once __DIR__ . '/../views/' . $vista . '.php';
        require_once __DIR__ . '/../views/layout/footer.php';
    }
}
models/Producto.php
models/Producto.php
<?php
class Producto
{
    // En este ejemplo los datos son estáticos (sin BD).
    // En un proyecto real aquí irían las consultas PDO.

    private static array $datos = [
        ['id' => 1, 'nombre' => 'Laptop Pro', 'precio' => 2_500_000, 'categoria' => 'Tecnología'],
        ['id' => 2, 'nombre' => 'Teclado RGB', 'precio' =>   180_000, 'categoria' => 'Tecnología'],
        ['id' => 3, 'nombre' => 'Silla Gamer', 'precio' =>   850_000, 'categoria' => 'Mobiliario'],
    ];

    // Devuelve todos los productos
    public static function todos(): array
    {
        return self::$datos;
    }

    // Busca uno por id; devuelve null si no existe
    public static function buscar(int $id): ?array
    {
        foreach (self::$datos as $p) {
            if ($p['id'] === $id) return $p;
        }
        return null;
    }
}
controllers/ProductoController.php
controllers/ProductoController.php
<?php
require_once __DIR__ . '/../models/Producto.php';

class ProductoController
{
    // Acción: listar todos
    public function index(): void
    {
        // 1. Pide datos al Model
        $productos = Producto::todos();

        // 2. Los pasa a la View (nada de HTML aquí)
        View::render('productos/lista', compact('productos'));
    }

    // Acción: mostrar uno
    public function show(): void
    {
        $id = (int) ($_GET['id'] ?? 0);
        $producto = Producto::buscar($id);

        if ($producto === null) {
            header('Location: index.php?c=home&a=notFound');
            exit;
        }

        View::render('productos/detalle', compact('producto'));
    }
}

Patrón clave: El Controller hace View::render('productos/lista', compact('productos')). La función compact() de PHP convierte la variable $productos en el array ['productos' => $productos]. Dentro de la view, extract() lo convierte de vuelta en $productos. Así el Controller y la View se comunican sin acoplarse.

Reglas de oro del MVC

✅ El Model nunca genera HTML

Solo trabaja con datos y lógica. Si ves echo "<div>" en un Model, algo está mal.

✅ La View nunca consulta la BD

Si ves new PDO() o $pdo->query() en una vista, algo está mal.

✅ El Controller solo coordina

No genera HTML ni escribe SQL. Su trabajo es recibir, llamar al Model, y pasarle el resultado a la View.

❌ La View no toma decisiones de negocio

Cálculos como $precio * 0.9 pertenecen al Model. La View solo formatea para mostrar.

Frameworks PHP que usan MVC

Una vez que domines MVC "a mano", el salto a un framework es mucho más fácil porque todos siguen el mismo patrón:

Laravel Symfony CodeIgniter Yii2 Zend / Laminas Phalcon CakePHP
Laravel específicamente implementa MVC con convenciones claras: los Models en app/Models/, los Controllers en app/Http/Controllers/ y las Views (llamadas Blade templates) en resources/views/. Si entiendes MVC desde cero, Laravel se aprende en días.
